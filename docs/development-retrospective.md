## 二、时间花在了哪里

### 2.1 各阶段时间分布（估算）

```
┌─────────────────────────────────┬──────────┬──────────────────────────────┐
│ 阶段                            │ 占比     │ 说明                         │
├─────────────────────────────────┼──────────┼──────────────────────────────┤
│ 1. 初始架构与核心功能搭建        │ ~15%     │ 最快的阶段                    │
│ 2. LLM Prompt 调优              │ ~30%     │ 最耗时 — 分类准确率反复调整    │
│ 3. 信用卡账单解析 (highlight)    │ ~20%     │ OCR+OpenCV+LLM 三层联调       │
│ 4. 分类一致性修复                │ ~20%     │ 同类商户 L1 不统一，反复 debug │
│ 5. 子文件夹/输出/UI 等边缘问题   │ ~10%     │ 路径、文件归档等工程细节       │
│ 6. 性能优化                     │ ~5%      │ 批量分类 + 并发 OCR            │
└─────────────────────────────────┴──────────┴──────────────────────────────┘
```

**核心结论：超过 70% 的时间花在了「LLM 输出质量调优」上（Prompt 调优 + 分类一致性 + 信用卡解析），而非传统意义上的"写代码"。**

### 2.2 具体耗时点剖析

#### 🔴 Prompt 调优（最大时间黑洞）

每次发现分类结果不对，流程是：
1. 跑一遍完整流程 → 看 Excel 输出 → 发现某些记录分类错误
2. 分析原因 → 修改 Prompt 中的规则
3. 再跑一遍 → 发现修了 A 又破坏了 B
4. 重复…

典型案例：
- **FRONTIER AI 被分为「交通费」**：因为地址含城市名 "San Francisco"，LLM 误判为交通类
- **Uber 有时是「美国-交通费」有时是「美国出差打车」**：L1 列表中两个类目含义重叠
- **团建费 vs 招待费 混淆**：火锅店发票放在"招待"子文件夹里，但文件名含"聚餐"触发了团建规则
- **港澳直通车600**：L2 写成了"港澳直通车600"而非"港澳直通车"

#### 🟡 信用卡 Highlight 解析

信用卡账单是整个项目中复杂度最高的模块，涉及三层协同：
- **OpenCV HSV 颜色检测** → 找到高亮区域坐标
- **RapidOCR** → 对高亮区域文字进行识别
- **DeepSeek LLM** → 结构化提取交易明细

Bug 来源：
- OCR 把商户名和金额分到不同行，LLM 拿不到完整交易
- 汇率说明行（"EXCHG RATE"）被当成独立交易
- 加拿大元消费误判为 CAD 而非折算后的 USD
- 高亮行只有日期+金额没有商户名，被直接丢弃
- 金额总和反复偏差（目标 $1452.42）

#### 🟠 分类一致性

LLM 的输出是概率性的。即使 Prompt 规则写得很清楚，同一个 Uber 在第一批分类中是「美国出差打车」，第二批可能变成「美国-交通费」。

解决方案的演进：
1. **v1: 硬编码规则** → `_normalize_l1()` 用关键词匹配（Uber → 美国出差打车）
2. **v2: 精简 L1 列表** → 删除重叠类目（美国-交通费 → 并入美国出差打车）
3. **v3: LLM 审核层** → 分类后再调一次 LLM 做全局 L1 一致性审核

每一步都是因为上一步仍然有新边界情况冒出来。

## 三、为什么会有这么多 Bug

### 3.1 LLM 本质上是概率模型

| 传统代码 | LLM 驱动 |
|----------|----------|
| 规则确定 → 输出确定 | 同一输入 → 输出可能不同 |
| 一次写对就不会错 | 对了 95% 还有 5% 边界情况 |
| 错误容易定位和复现 | 错误难以稳定复现 |
| 单元测试覆盖即可 | 需要大量真实数据回归 |

这意味着：每增加一条 Prompt 规则去修 Bug A，都可能干扰 Bug B 的已有修复。Prompt 工程本质上是在一个高维空间中调参。

### 3.2 数据多样性超出预期

初始 Prompt 基于已知的样本设计，但真实发票千变万化：
- 子文件夹命名带金额符号（`招待 ¥1,771/`）
- 文件名中嵌入金额（`港澳直通车600.pdf`）
- 信用卡账单格式因银行不同而异
- OCR 质量不一致（手机拍照 vs 扫描件）

### 3.3 多层管道放大误差

```
OCR 识别 → LLM 解析 → LLM 分类 → 后处理规范化 → 输出
```

每一层都有自己的误差空间。OCR 漏字 → LLM 解析缺字段 → 分类错误 → 规范化无法修复。管道越长，端到端调试越困难。

### 3.4 缺少前期测试数据

项目一开始只有 reference 里的 16 个样本文件。许多 bug 直到跑真实数据才暴露：
- 信用卡 highlight 解析 → 需要真实高亮 PDF 才能验证
- 子文件夹处理 → 只有真实文件夹结构才能触发
- 分类一致性 → 需要 50+ 条记录才能看出 L1 不统一

## 四、如何改进 — 减少 Debug 与交互成本

### 4.1 📋 更完整的初始 Prompt

**问题**：原始 PROMPT.md 描述了"要做什么"，但缺少"边界情况长什么样"。

**改进**：
- 在初始 Prompt 中直接附上 **5-10 个边界 case 的期望输出**
- 明确列出"不要做什么"（负面示例同样重要）
- 对 L1 分类列表做 **清晰的互斥定义**，不留模糊空间
- 提前说明子文件夹、文件名中的金额、特殊字符等情况

### 4.2 🧪 先建测试，再写功能

**问题**：每次发现 bug 都是在 UI 上人工跑一遍才看到，反馈循环太慢。

**改进**：
- 对每种文件类型（普通发票 / 信用卡全部 / 信用卡高亮）各建一个 **golden test**
- Golden test = 固定输入 → 期望输出（金额、记录数、L1 分类）
- 每次改 Prompt 或代码后自动跑测试，**3 秒出结果**而不是等 LLM 返回
- 对于 OCR 和解析层，用 **mock LLM response** 做快速单元测试

### 4.3 🎯 Prompt 与代码职责分离

**问题**：经常在 Prompt 和 Python 代码之间来回搬规则，导致规则分散在两处。

**改进**：
- **Prompt 只负责理解语义**（"这是什么商户"、"这是什么费用"）
- **代码负责强约束**（L1 必须在白名单内、同类商户统一映射）
- 这样 Prompt 调优不会破坏已有的强规则，代码变更也不影响语义理解

### 4.4 📦 缩短反馈循环

| 当前流程 | 改进后 |
|----------|--------|
| 改代码 → 启动 Streamlit → 选文件夹 → 等 OCR → 等分类 → 看结果 | 改代码 → `python test_quick.py` → 3 秒出结果 |
| 每次全量跑 ~95 笔 | 只跑变更相关的 5 笔 |
| 看 Excel 才发现问题 | 测试脚本自动比对 |

具体措施：
- 缓存 OCR 结果，Prompt 调优时跳过 OCR 直接从缓存走分类
- 建立 `test_classification_regression.py`，包含 20 个期望分类 case
- 每次 Prompt 改动前后自动 diff 分类结果

### 4.5 🔄 分阶段验收

**问题**：一次给 AI 生成整个项目 → 只在最后验收 → 发现大量问题 → 进入漫长的修 bug 循环。

**改进**：
```
Phase 1: 只做 OCR + 结构化解析 → 验收准确率达标
Phase 2: 加分类 → 验收分类准确率
Phase 3: 加汇率/去重/导出 → 验收完整流程
Phase 4: 加信用卡处理 → 验收复杂场景
```

每个阶段交付可测试的最小功能，验收通过后再进入下一阶段。这样 bug 出现时能精准定位在当前阶段，而不是在 10 个模块中找问题。

### 4.6 📝 给 AI 更好的上下文

- 每轮交互前，把上一轮的**实际输出**（而非描述性文字）贴给 AI
- 用截图或日志而非口述来报告 bug → 减少信息损失
- 维护一个 `KNOWN_ISSUES.md`，让 AI 知道哪些问题已解决、哪些是已知限制

## 五、关键教训总结

| # | 教训 | 原则 |
|---|------|------|
| 1 | LLM 分类准确率不是 0 就是 100 的问题，而是一个**持续调优**的过程 | 预留调优时间，不要假设 Prompt 一次就对 |
| 2 | **重叠的分类类目**是最大的 bug 源头 | 设计分类体系时确保互斥 |
| 3 | 文件名/路径中的信息比 OCR 文字更可靠 | 优先信任结构化元数据 |
| 4 | 信用卡账单 ≠ 普通发票，复杂度高一个量级 | 作为独立模块设计和测试 |
| 5 | 每增加一层 LLM 调用都会增加不确定性 | 能用代码实现的不要交给 LLM |
| 6 | 没有自动化测试的 Prompt 改动 = 盲改 | 先建 golden test 再调 Prompt |
| 7 | 初始 Prompt 中的**负面示例**和原始 Prompt 同样重要 | 告诉 AI "不要做什么" |

## 六、如果重来一次

如果从头做这个项目，我会：

1. **花两倍时间写初始 Prompt**，包括 20 个边界 case 的期望输出
2. **一开始就建好自动化测试框架**，每种文件类型至少 3 个 golden test
3. **分 4 个阶段交付**，每个阶段独立验收
4. **把 L1 分类列表做到互斥无歧义**，不留任何重叠类目
5. **缓存中间结果**，调 Prompt 时不用每次从 OCR 开始跑
6. **把信用卡模块单独设计和验证**，不和普通发票混在一起调试
